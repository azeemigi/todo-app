# Implementation Plan: TODO Filtering, Search, and Sort

**Branch**: `002-todo-filtering-search-sort` | **Date**: 2026-06-08 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/002-todo-filtering-search-sort/spec.md`

## Summary

Extend the existing TODO application with list controls for status filtering (All / Active / Completed), case-insensitive text search across title and description, and sort ordering by creation date or title (ascending/descending). The API gains optional query parameters on `GET /api/todos`; the Angular frontend makes URL query params the single source of truth for view state, enabling persistence across refresh, deep links, and browser history navigation. Stale API responses from rapid control changes are discarded via a generation counter in the service.

## Implementation Strategy

### Backend (US1–US3 API layer)

1. Define `TodoStatus`, `SortBy`, `SortDir` enums with lowercase API string values for Spring MVC binding.
2. Add `findAll(TodoStatus, String q, SortBy, SortDir)` to `TodoService` using Java streams: filter by status, filter by `q` substring (case-insensitive, trim whitespace), sort by field + direction.
3. Update `TodoController.getAllTodos()` to accept `@RequestParam` params with enum types and default values.
4. Add `MethodArgumentTypeMismatchException` handler in `GlobalExceptionHandler` to return the existing `{"errors":[...]}` shape.

**Test approach (TDD)**:
- `TodoServiceTest`: unit-test each filter combination and sort option against a seeded in-memory map.
- `TodoControllerTest` (`@WebMvcTest`): test valid param combinations (200), invalid enum values (400 with correct error shape), and no-param backward compatibility.

### Frontend (US1–US5)

1. Add `TodoFilter` interface to `todo.model.ts`.
2. Extend `TodoService.loadTodos(filter?: TodoFilter)`: build `HttpParams` from filter fields (omit absent/empty values), add generation counter for stale response protection, surface API errors via existing error signal.
3. Create `TodoListControlsComponent` (dumb): `@Input()` for current filter state, `@Output()` events for each control change. No Router dependency. Tests use only input/output bindings.
4. Modify `TodoListComponent` (smart): inject `ActivatedRoute` and `Router`; convert `queryParamMap` to a signal with `toSignal()`; derive `status`, `q`, `sortBy`, `sortDir` as `computed()` signals with fallback defaults; use `effect()` to call `svc.loadTodos()` on param changes; update URL on control `@Output()` events using `router.navigate([], { queryParamsHandling: 'merge' })`.

**Test approach (TDD)**:
- `TodoListControlsComponent` spec: verify `@Output()` emissions for each control interaction.
- `TodoListComponent` spec: verify URL hydration on init, URL update on control change, and that `loadTodos()` is called with correct params.
- `TodoService` spec: verify generation counter discards stale responses; verify `HttpParams` built correctly for each filter combo.

### CRUD integration (FR-017)

After each CRUD operation completes, `TodoService` calls `loadTodos()` with the current `TodoFilter` (received from `TodoListComponent`). This ensures the post-CRUD refresh respects active filters. `TodoListComponent` passes the current filter to the service rather than the service holding it independently.

## Key Design Decisions

### Generation counter (stale response protection)

```
private requestGen = 0;

loadTodos(filter?: TodoFilter): void {
  const gen = ++this.requestGen;
  // ... http.get().subscribe({ next: todos => { if (gen !== this.requestGen) return; ... } })
}
```

Each `loadTodos()` call captures the current generation. The subscribe callback no-ops if a newer call has since incremented `requestGen`. This is deterministic, requires no RxJS operators, and is trivially testable by calling `loadTodos()` twice and verifying only the second response is applied.

### URL as source of truth

`TodoListComponent` never stores filter state in a local signal. The `queryParams` signal (from `toSignal(route.queryParamMap)`) IS the state. Computed signals derive display values from it. `effect()` triggers API calls. `router.navigate()` writes new state. This means back/forward navigation is automatic — the Router's `queryParamMap` emits on history change, which invalidates the computed signals, which triggers the `effect()`, which fetches the new list.

### Invalid URL params → silent fallback in frontend

The frontend validates each URL param value on read and replaces invalid enum strings with the default. This prevents `400` errors being surfaced to users who arrive via a stale or manually-edited link, while the API still enforces `400` for programmatic consumers.
